# Аудит логики FinLayer — отчёт

- **Дата:** 2026-06-01
- **Issue:** [#11 «We need to check all the logic»](https://github.com/xlabtg/FinLayer/issues/11)
- **PR:** [#12](https://github.com/xlabtg/FinLayer/pull/12)
- **Объём:** полный сквозной разбор бизнес‑логики, безопасности, корректности денежных расчётов и устойчивости сервиса по всем доменам монорепозитория (`swap`, `payments`, `earn`, `wallet`, `affiliate`, `analytics`, `auth`, провайдеры, инфраструктура, SDK).

## Резюме

Проведён ручной аудит исходного кода с упором на поиск **корневых причин**, а не симптомов. Каждая находка подтверждена прямой цитатой кода с указанием `файл:строки`. Кандидаты, оказавшиеся ложными срабатываниями, вынесены в отдельный раздел с обоснованием, чтобы команда не тратила время на их «исправление».

Итог: **5 критичных**, **7 высоких**, **8 средних** и **5 низких** подтверждённых находок. По каждой заводится отдельный профессиональный issue с метками `severity:*`, `domain:*`, типом (`security`/`vulnerability`/`bug`) и этапом внедрения (`phase:*`).

### Этапы внедрения

- **Этап 1 (`phase:1-critical-fixes`)** — критичные проблемы безопасности и финансов, эксплуатируемые удалённо или приводящие к денежному ущербу/недоступности. Чинятся первыми.
- **Этап 2 (`phase:2-hardening`)** — корректность денежных расчётов, устойчивость к гонкам, конфигурация безопасности.
- **Этап 3 (`phase:3-tech-debt`)** — технический долг, мёртвый код, упрочнение валидации, наблюдаемость.

## Критичные находки (Этап 1)

### C1. Вебхук свопа без аутентификации → подмена статуса любой транзакции
- **Где:** `modules/swap/routes.ts:208‑239`
- **Корневая причина:** маршрут `POST /v1/swap/webhook/:id` зарегистрирован **без** `preHandler`/аутентификации и **без** проверки подписи провайдера. Параметр `:id` типизирован как обычная `string` (не `uuid`), а тело обрабатывается как `UPDATE transactions SET status = ${status} ... WHERE id = ${id}` без фильтра по домену/провайдеру.
- **Воздействие:** любой неаутентифицированный клиент может выставить произвольный статус (`completed`, `refunded`, `failed`, …) **любой** транзакции в едином реестре `transactions` — включая платежи и earn, не только свопы. Это прямой вектор финансового мошенничества и порчи данных.
- **Рекомендация:** проверять HMAC‑подпись провайдера (как сделано в платежах), валидировать `:id` как UUID, фильтровать обновление по домену/провайдеру и применять допустимые переходы статусов (см. C‑payments state machine, H3).

### C2. Скан API‑ключей `LIMIT 20` без `ORDER BY` → блокировка аутентификации и CPU‑DoS через bcrypt
- **Где:** `modules/auth/service.ts:103‑123`
- **Корневая причина:** `key_prefix` одинаков для всех ключей одного типа (`fl_live` / `fl_test`, см. `packages/utils/src/index.ts:74‑78`). Запрос `SELECT * FROM api_keys WHERE key_prefix = ${prefix} ... LIMIT 20` **без `ORDER BY`** возвращает произвольные 20 строк, после чего по каждой выполняется `bcrypt.compare`.
- **Воздействие:**
  1. **Блокировка аутентификации:** как только активных боевых ключей становится больше 20, валидный ключ может не попасть в выборку → недетерминированный отказ в аутентификации легитимным клиентам.
  2. **CPU‑DoS:** запрос `fl_live_<мусор>` заставляет сервер выполнять до 20 `bcrypt.compare` (намеренно дорогих) на каждый запрос — дешёвый способ исчерпать CPU.
- **Рекомендация:** хранить и искать по уникальному, индексируемому идентификатору ключа (например, `key_id`, передаваемый в самом ключе), выполнять ровно один `bcrypt.compare`. Убрать `LIMIT 20`‑эвристику.

### C3. TOCTOU идемпотентности → двойной вызов провайдера/двойное списание
- **Где:** `modules/swap/service.ts:309‑422`, `modules/payments/service.ts:84‑213`, `modules/earn/service.ts:99‑338`
- **Корневая причина:** во всех трёх доменах порядок операций такой: (1) `SELECT` по `idempotency_key` → (2) **вызов провайдера** (`provider.executeSwap` / `createInvoice` / `deposit`/`withdraw`), создающий внешний побочный эффект → (3) `INSERT` в `transactions`. Между шагами (1) и (3) нет блокировки. Колонка `transactions.idempotency_key` имеет `UNIQUE` (`apps/api/src/db/migrations/001_initial_schema.sql:129`), поэтому дублирующий `INSERT` упадёт — но **уже после** того, как провайдер выполнил операцию.
- **Воздействие:** два одновременных запроса с одинаковым `idempotency_key` оба проходят `SELECT` (строки ещё нет), оба вызывают провайдера (две реальные обменные операции/инвойса/депозита), затем один `INSERT` выигрывает, а второй падает на нарушении уникальности → **осиротевшая/задвоенная операция у провайдера**, деньги ушли дважды, а в реестре только одна запись. Идемпотентность по сути не работает при конкурентных ретраях.
- **Рекомендация:** резервировать ключ идемпотентности в БД (INSERT со статусом `pending`) **до** вызова провайдера, либо оборачивать в `INSERT ... ON CONFLICT DO NOTHING` + advisory‑lock, и только после успешной резервации обращаться к провайдеру.

### C4. `SIGTERM` создаёт новый экземпляр приложения и закрывает его, а не работающий сервер
- **Где:** `apps/api/src/index.ts:223‑227`
- **Корневая причина:** обработчик `process.on('SIGTERM', async () => { const app = await buildApp(); await app.close(); process.exit(0); })` собирает **новый** экземпляр Fastify и закрывает его, а реально запущенный сервер (`start()` → `app.listen`) никогда не закрывается.
- **Воздействие:** graceful shutdown не работает: при остановке/деплое не закрываются пул БД и in‑flight запросы работающего сервера, возможна потеря соединений и незавершённых транзакций. Дополнительно тратится время/ресурсы на лишнюю сборку приложения.
- **Рекомендация:** держать ссылку на запущенный экземпляр и в обработчиках `SIGTERM`/`SIGINT` закрывать именно его (`await runningApp.close()`).

### C5. Гонка выплат партнёрам при нескольких инстансах → двойная выплата
- **Где:** `modules/affiliate/scheduler.ts:99‑192`
- **Корневая причина:** `runOnce` защищён только in‑memory флагом `this.running` (работает лишь в пределах одного процесса). `createPayout` (строки 143‑192) делает `SELECT ... FROM revenue_events WHERE distributed_at IS NULL` **без `FOR UPDATE`/блокировки строк**, а затем в одной транзакции вставляет `affiliate_payout_items` и проставляет `distributed_at`. Первичный ключ `affiliate_payout_items (payout_id, revenue_event_id)` (`003_wallet_phase4.sql:68`) **не** запрещает попадание одного `revenue_event_id` в разные `payout_id`.
- **Воздействие:** при горизонтальном масштабировании (несколько инстансов/планировщиков) два запуска читают один и тот же набор нераспределённых событий и создают **две выплаты** по ним → переплата партнёрам. Комментарий в коде, утверждающий «Re‑runs therefore cannot double‑pay», не соответствует действительности.
- **Рекомендация:** `SELECT ... FOR UPDATE SKIP LOCKED`, либо `UNIQUE` на `affiliate_payout_items.revenue_event_id` (см. M4), либо распределённый лок/advisory‑lock на запуск планировщика.

## Высокие находки (Этап 2)

### H1. Атрибуция партнёрской выручки без проверки связи → self‑referral мошенничество
- **Где:** `modules/auth/plugin.ts:44‑50`, `modules/swap/revenue.ts:31‑73`, `modules/affiliate/service.ts:142‑147`
- **Корневая причина:** `affiliate_id` берётся из тела запроса (валидируется только как UUID) либо пробрасывается из query/заголовка `x-affiliate-id` в auth‑плагине **без какой‑либо проверки**. `createRevenueEvent` затем кредитует `affiliates.total_earned` на указанный `affiliate_id`. Нет проверки, что плательщик не является владельцем этого партнёрского профиля. Функция `validateAffiliateId` определена, но **нигде не вызывается** (мёртвый код), и даже её вызов проверял бы лишь существование (FK это и так гарантирует), а не отсутствие self‑referral.
- **Воздействие:** любой пользователь может создать партнёрский профиль и указывать собственный `affiliate_id` на своих же транзакциях, накручивая комиссию, которую затем выплатит планировщик (C5).
- **Рекомендация:** связывать атрибуцию с зафиксированным фактом перехода по партнёрской ссылке (а не с произвольным значением из запроса), запретить self‑referral (плательщик ≠ владелец affiliate), задействовать/усилить `validateAffiliateId`.

### H2. Денежные расчёты на `parseFloat`/float вместо точной арифметики
- **Где:** `modules/swap/revenue.ts:67,92`, `modules/affiliate/scheduler.ts:110,168,123`, `modules/swap/service.ts` (расчёт `platformFee`), `modules/providers/changenow/adapter.ts:139`; при этом точные `addNumericStrings`/`multiplyNumericStrings` в `packages/utils/src/index.ts:24‑50` **не используются**.
- **Корневая причина:** комиссии и доли считаются через `parseFloat(...) * ratio` с последующим `.toFixed(8)`, что вносит ошибки округления double для денежных величин.
- **Воздействие:** систематические расхождения в суммах комиссий/выплат, потенциально накапливающиеся; реестр расходится с реальными деньгами.
- **Рекомендация:** перевести все денежные операции на BigInt/decimal‑арифметику (уже имеющиеся утилиты или библиотеку фикс‑точности), хранить суммы как `numeric`.

### H3. Обновление статуса платежа без машины состояний → возможен «откат» статуса
- **Где:** `modules/payments/service.ts:437‑492`
- **Корневая причина:** `applyStatusUpdate` безусловно мапит статус провайдера через `INVOICE_TO_TX_STATUS` и записывает его, не проверяя допустимость перехода. Кроме того, выручка считается по `txRow.amount` (запрошенная сумма), а не по фактически оплаченной.
- **Воздействие:** повторный/устаревший вебхук может перевести уже завершённый инвойс обратно в `pending`/`processing`; некорректный учёт выручки при частичной оплате.
- **Рекомендация:** ввести явную машину состояний (терминальные статусы неизменяемы), считать выручку по фактически оплаченной сумме.

### H4. `wallet.getBalance` возвращает нативный баланс, игнорируя актив адреса
- **Где:** `modules/wallet/service.ts:155‑183`
- **Корневая причина:** метод всегда вызывает `provider.getNativeBalance(...)` и возвращает `result.asset` (нативный), игнорируя `row.asset`. Для токенов (например, USDC на Ethereum) вернётся баланс ETH, а не токена.
- **Воздействие:** неверные балансы для всех неносительных активов — критично для финансового продукта.
- **Рекомендация:** маршрутизировать на запрос баланса токена (по контракту актива) когда `asset` не нативный; для нативных — как сейчас.

### H5. `rateLimitStore` — неограниченная in‑memory Map, не очищается
- **Где:** `modules/auth/service.ts:43` (объявление), `checkRateLimit` (используется в `validateApiKey`)
- **Корневая причина:** счётчики рейт‑лимита хранятся в модульной `Map`, которая чистится только при отзыве ключа. Записи накапливаются неограниченно.
- **Воздействие:** утечка памяти при большом числе ключей; кроме того, лимиты считаются per‑process и не работают корректно при нескольких инстансах.
- **Рекомендация:** TTL‑очистка/скользящее окно с истечением, либо вынос в Redis для распределённого лимита.

### H6. ChangeNOW `executeSwap` отправляет `fromAmount: 0, toAmount: 0` → котировка не соблюдается
- **Где:** `modules/providers/changenow/adapter.ts:169‑194`
- **Корневая причина:** при исполнении свопа в провайдер уходят нулевые суммы, что переводит обмен в режим плавающего курса ChangeNOW; ранее показанные пользователю `to_amount`/`rate` не гарантируются.
- **Воздействие:** пользователь получает сумму, отличную от котировки; расхождение между обещанным и фактическим результатом, споры, потери.
- **Рекомендация:** передавать реальную сумму и использовать фиксированный курс (`fixed-rate`), сверять полученную котировку с сохранённой.

### H7. CORS `origin: '*'` (fallback) вместе с `credentials: true`
- **Где:** `apps/api/src/index.ts:60‑63`
- **Корневая причина:** `origin: process.env['CORS_ORIGINS']?.split(',') ?? '*'` при `credentials: true`. Если переменная не задана — разрешены любые источники с передачей кредами.
- **Воздействие:** ослабление защиты от межсайтовых запросов с учётными данными; небезопасная конфигурация по умолчанию.
- **Рекомендация:** запретить wildcard при `credentials: true`; требовать явный allow‑list, без безопасного дефолта — падать/предупреждать.

## Средние находки (Этапы 2–3)

### M1. Open redirect через партнёрский редирект `/r/:code`
- **Где:** `modules/affiliate/routes.ts:128‑156`, `modules/affiliate/service.ts:129‑137`
- **Корневая причина:** `reply.redirect(302, targetUrl)`, где `targetUrl` валидируется только `z.string().url()` при создании ссылки. Любой валидный URL (включая внешний фишинговый) допустим.
- **Воздействие:** открытый редирект → фишинг/репутационный ущерб (требует `affiliate:write` для создания ссылки, что снижает остроту до medium).
- **Рекомендация:** allow‑list доменов назначения либо страница‑интерстишал с подтверждением.

### M2. Дублирующая регистрация партнёрских маршрутов в корне
- **Где:** `apps/api/src/index.ts:197` и `:204`
- **Корневая причина:** `affiliateRoutes` регистрируется и под префиксом `/v1/affiliate`, и в корне (для редиректа `/r/:code`). В результате **весь** роутер партнёрки (`/link`, `/stats`, `/payouts`, `/payouts/run`) дублируется в корне без префикса. Скоупы по‑прежнему проверяются (не обход авторизации), но это непреднамеренное расширение поверхности.
- **Рекомендация:** вынести только публичный редирект в отдельный мини‑роутер, регистрируемый в корне; остальное — только под `/v1/affiliate`.

### M3. Небезопасный парсинг `providerQuoteId` в ChangeNOW с дефолтами `btc`/`eth`
- **Где:** `modules/providers/changenow/adapter.ts:172‑173`
- **Корневая причина:** `fromCurrency`/`toCurrency` извлекаются из `providerQuoteId.split('_')` с дефолтами `'btc'`/`'eth'` при отсутствии частей.
- **Воздействие:** при неожиданном формате идентификатора своп может уйти по неверной валютной паре.
- **Рекомендация:** хранить пару валют явно (в БД котировки), не парсить из строки‑идентификатора.

### M4. Нет `UNIQUE` на `affiliate_payout_items.revenue_event_id`
- **Где:** `apps/api/src/db/migrations/003_wallet_phase4.sql:64‑68`
- **Корневая причина:** PK `(payout_id, revenue_event_id)` допускает одно и то же событие выручки в разных выплатах.
- **Воздействие:** отсутствует БД‑уровневая защита от двойной выплаты (усиливает C5).
- **Рекомендация:** добавить `UNIQUE (revenue_event_id)` (или частичный уникальный индекс) — гарантия «событие оплачивается ровно один раз».

### M5. Нет индекса под горячий запрос планировщика по `revenue_events`
- **Где:** `apps/api/src/db/migrations/001_initial_schema.sql` (раздел `revenue_events`), запрос в `modules/affiliate/scheduler.ts:106‑118,144‑150`
- **Корневая причина:** фильтрация по `affiliate_id` + `distributed_at IS NULL` без поддерживающего индекса.
- **Воздействие:** полное сканирование таблицы по мере роста выручки → деградация выплат.
- **Рекомендация:** частичный индекс `revenue_events (affiliate_id) WHERE distributed_at IS NULL`.

### M6. `sql.unsafe` со строковой интерполяцией в analytics (латентный SQLi)
- **Где:** `modules/analytics/service.ts:134,172,212,258,295` (и `sinceClause`, `bucket`)
- **Корневая причина:** запросы строятся через `sql.unsafe(...)` с интерполяцией `affiliateId`/выражений. Сейчас `affiliateId` приходит как доверенный серверный UUID (`getAffiliateDashboard(profile.id, ...)`), `period` — enum, поэтому **активной** инъекции нет. Но паттерн обходит параметризацию и в одном рефакторинге от уязвимости.
- **Воздействие:** высокий риск SQL‑инъекции при любом будущем изменении источника параметров.
- **Рекомендация:** использовать параметры `sql.unsafe(query, params)` или теговые шаблоны; не интерполировать значения в строку.

### M7. Глобально отключён CSP в helmet
- **Где:** `apps/api/src/index.ts:56‑58`
- **Корневая причина:** `contentSecurityPolicy: false` ради Swagger UI отключает CSP для всего приложения.
- **Воздействие:** ослабление защиты от XSS на всех маршрутах, не только на `/docs`.
- **Рекомендация:** включить CSP с отдельной, более мягкой политикой только для маршрута документации.

### M8. Неоднозначный порядок миграций (три файла `003_*`)
- **Где:** `apps/api/src/db/migrations/003_earn_providers.sql`, `003_payments.sql`, `003_wallet_phase4.sql`
- **Корневая причина:** три миграции с одинаковым префиксом `003`; порядок применения зависит от сортировки имён, что хрупко при наличии межтабличных зависимостей.
- **Рекомендация:** уникальная монотонная нумерация версий миграций.

## Низкие находки (Этап 3)

### L1. Мёртвый код `validateAffiliateId`
- **Где:** `modules/affiliate/service.ts:142‑147` — определён, нигде не вызывается (см. H1).

### L2. Слабая валидация криптоадреса (`length >= 10`)
- **Где:** `packages/utils/src/index.ts:61‑63` — `isValidCryptoAddress` проверяет только длину строки, без формата сети.
- **Рекомендация:** валидация по сети (EVM checksum, base58 для BTC и т. п.).

### L3. Дублирующий/мёртвый пул соединений БД (CLI‑only)
- **Где:** `apps/api/src/db/client.ts` создаёт отдельный пул, используемый только CLI миграций; серверный пул живёт в `apps/api/src/plugins/database.ts`. Это не «исчерпание соединений», а дублирование/мёртвый код.
- **Рекомендация:** документировать назначение или переиспользовать единую фабрику.

### L4. SDK поддерживает заголовок `Idempotency-Key`, но `swap.execute` его не шлёт
- **Где:** `packages/sdk/src/client.ts:50‑68`, `packages/sdk/src/modules/swap.ts:49‑55` — ключ идемпотентности фактически едет в теле запроса (сервер читает его из body), заголовок не отправляется. Косметика/несогласованность, не уязвимость.
- **Рекомендация:** привести к единообразию (либо слать заголовок, либо убрать его поддержку).

### L5. `last_used_at` обновляется fire‑and‑forget без обработки ошибок
- **Где:** `modules/auth/service.ts:133‑135` — `void this.sql\`UPDATE ... last_used_at ...\`` без `.catch`, возможен необработанный rejection.
- **Рекомендация:** добавить обработку ошибок/логирование, либо батчить обновления.

## Ложные срабатывания (исключены из issues с обоснованием)

Чтобы команда не тратила усилия, ниже — кандидаты, проверенные и **опровергнутые**:

1. **«SDK не шлёт `Idempotency-Key` → идемпотентность сломана (critical)»** — опровергнуто. Ключ передаётся в теле запроса, сервер читает его оттуда (`idempotency_key`). См. L4 — это лишь косметика.
2. **«Двойной пул соединений → исчерпание коннектов»** — опровергнуто. `apps/api/src/db/client.ts` используется только CLI миграций; серверный пул отдельный (`plugins/database.ts`). См. L3 (мёртвый/дублирующий код, не утечка).
3. **«APY Compound завышен в ~100 раз»** — опровергнуто. И Aave, и Compound адаптеры приводят ставку к процентам согласованно (`aaveLiquidityRateToApy` → `rate * SECONDS_PER_YEAR * 100`).
4. **«Вебхук MoonPay/платежей fail‑open (принимает без подписи)»** — опровергнуто. `verifyWebhook` (`modules/providers/moonpay/adapter.ts:112‑151`) реализует `signatureValid = secret ? verifyHmacSha256(...) : false` и `handleWebhook` отклоняет неподписанные запросы (`modules/payments/service.ts:298‑302`) — поведение **fail‑closed**, с `timingSafeEqual`. Это, наоборот, контрастирует с C1 (вебхук свопа вообще без подписи).
5. **«Analytics — активная SQL‑инъекция (critical)»** — переклассифицировано в M6. `affiliateId` сейчас всегда доверенный серверный UUID, `period` — enum; активной инъекции нет, но паттерн `sql.unsafe`+интерполяция опасен и подлежит исправлению.

## Методология

- Чтение исходников по доменам с трассировкой потоков данных от маршрута до БД и до вызова провайдера.
- Проверка схемы БД (`apps/api/src/db/migrations/*.sql`) для оценки реального «радиуса поражения» (наличие/отсутствие `UNIQUE`, FK, индексов).
- Разделение находок на подтверждённые (с точными цитатами) и ложные срабатывания (с опровержением).
- Приоритизация по эксплуатируемости и финансовому/операционному воздействию; распределение по этапам внедрения.

Каждая находка из этого отчёта оформляется отдельным issue в репозитории с соответствующими метками и ссылкой на этот документ.
